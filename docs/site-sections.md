## Тема фону (scroll-driven, кожні 2 розділи)

Фон усієї сторінки перемикається dark ⇄ light залежно від того, який розділ перетинає середину вʼюпорта під час скролу (логіка — [BaseLayout.astro](../src/layouts/BaseLayout.astro), `LIGHT_IDS`/`ZONE_IDS`). Групи по 2 розділи, починаючи з темної:

`Hero + Регуляторний контекст` (темна) → `Можливості + Інтеграції` (світла) → `Реалізація + Валідація` (темна) → `Досвід та відповідальність + FAQ` (світла) → `Наступний крок` (темна, останній розділ — непарний залишок).

Компоненти написані на design-токенах (`--ink`, `--ink-soft`, `--on-dark`, ...), які автоматично інвертуються через `body.theme-dark`/`body.theme-light` в [global.css](../src/styles/global.css) — тому будь-який розділ технічно можна перенести в іншу групу без переписування компонента. Виняток — елементи з хардкодженими кольорами (наприклад мокап у Валідації, стрічка ERP-джерел, CTA), для яких у `global.css` є явні `body.theme-light .selector {...}` оверрайди на випадок, якщо розділ опиниться у світлій групі.

## Нотатки

- Розділи 3 («Можливості») та 7 («Досвід та відповідальність») мають `id`, але не мають пункту в навігації Header ([Header.astro](../src/components/Header.astro)) — за потреби додати окремим завданням.
- Усі `id` вже використовуються як якорі навігації (`href="#..."`) і збігаються з назвою розділу по суті (не завжди дослівно з eyebrow) — саме ці `id` слід використовувати як стабільні посилання на розділ у майбутніх задачах.
- Коли даєш завдання по конкретному розділу — можна називати його за eyebrow («розділ Валідація», «розділ Реалізація») або за `id` (`#validation`, `#how-it-works`) — обидва варіанти однозначно мапляться на компонент через цю таблицю.
